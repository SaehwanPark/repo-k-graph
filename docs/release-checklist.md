# Release Checklist

Use this checklist when preparing an `rkg` release from the public
`SaehwanPark/repo-k-graph` repository.

## Preflight

- Confirm `SPEC.md`, `ARCHITECTURE.md`, `CHANGELOG.md`, `README.md`, and
  `docs/user-manual.md` describe the same completed state.
- Confirm `[workspace.package] version` in `Cargo.toml` matches the release
  being tagged.
- Run:

```sh
cargo fmt --all -- --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
cargo package --workspace --allow-dirty --list
```

The `cargo package --workspace --allow-dirty --list` check verifies publishable
crate contents, binary names, included generated parser sources, license
metadata, README availability where configured, and package metadata.

## GitHub Release

After validation and explicit approval, create a version tag in the public
repository:

```sh
git tag v1.0.12
git push origin v1.0.12
```

The release workflow builds and uploads these binary assets:

- `rkg-linux-x86_64`
- `rkg-macos-aarch64`
- `rkg-windows-x86_64.exe`

It also uploads shell completions:

- `rkg.bash`
- `_rkg`
- `rkg.fish`

After the workflow finishes, verify the GitHub Release contains all assets and
that generated release notes match `CHANGELOG.md`.

## Homebrew

The Homebrew formula should point at the macOS release asset or a source archive
for the tagged release and must include an updated `sha256`.

Formula update flow:

```sh
curl -L -o rkg-macos-aarch64 \
  https://github.com/SaehwanPark/repo-k-graph/releases/download/v1.0.12/rkg-macos-aarch64
shasum -a 256 rkg-macos-aarch64
```

Update the formula `url`, `version`, and `sha256`, then validate locally:

```sh
brew audit --new --formula rkg
brew install --build-from-source ./Formula/rkg.rb
rkg --help
```

If the formula installs completions, verify bash, zsh, and fish completion files
are installed from the release assets or generated by the packaged binary.

## Cargo Install

Crates.io publishing is not part of the current public repository and binary
release flow. Before publishing any crate or documenting `cargo install`, confirm
packaging from a clean checkout and get explicit approval:

```sh
cargo package --workspace --allow-dirty --list
```

For a local install smoke test:

```sh
cargo install --path crates/rkg-cli --bin rkg --locked
rkg --help
```

Do not publish crates if generated parser sources, fixture files required by
tests, license metadata, repository metadata, or the `rkg` binary target are
missing from the package.

## Post-release

- Regenerate command docs if CLI help changed:

```sh
./scripts/gen_command_ref.sh
```

- Check `docs/user-manual.md` still links to:
  - `docs/release-checklist.md`
  - `docs/schema-reference.md`
  - `docs/language-adapter-guide.md`
- Record any deferred distribution work in `SPEC.md` Future rather than changing
  release behavior opportunistically.
